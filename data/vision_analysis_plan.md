# 이벤트 데이터 후처리 및 Vision LLM 분석 가공 파이프라인 (향후 계획)

본 계획은 기존 스크래퍼들이 성공적으로 수집한 CSV 데이터와 다운로드된 이벤트 이미지(썸네일, 디테일 이미지)를 바탕으로, 사전에 정의하신 **신규 헤더 구조**에 맞게 데이터를 정형화/가공하는 프로세서(Processor) 개발안입니다. 현재는 단순 헤더 분류만 적용되어 있으며, 향후 AI 도입 시 아래 계획을 따릅니다.

## ⚠️ User Review Required

이미지 내부 텍스트(조건, 카테고리, 혜택 등)를 읽어 정형화하려면 기존 DOM 스크래퍼 로직만으로는 불가능하며 **Vision 기반의 AI 모델(LLM)**을 사용하여 이미지를 분석해야 합니다. 이를 위해 어떤 형태의 AI API를 사용하실지에 대한 결정(예: **OpenAI GPT-4o, Google Gemini 1.5 Pro, Anthropic Claude 3.5 Sonnet** 등)과 해당 **API Key**가 필요합니다.

## 제안하는 변경 사항

### 1. `processor.py` 연동
- **이미지 로드 및 Vision 처리**: CSV의 `detail_images` 컬럼과 `thumbnail_url` 컬럼에 명시된 이미지 파일에 액세스 한 뒤, 선택된 LLM의 Vision API(예: OpenAI `gpt-4o`)에 해당 이미지들을 전달합니다.

### 2. 가공 목표 (Output Schema)
LLM에 **Structured Output(JSON)** 형태로 데이터를 반환하도록 프롬프트를 설계합니다.
  1. `분류`: (자체이벤트 / 제휴이벤트 중 1개 선택)
  2. `일반/특별`: ("일반"으로 고정 후 별도 검토)
  3. `제휴사`: (제휴사가 존재할 시 회사명 추출, 없을 시 빈칸)
  4. `프로모션명`: (기존 스크래핑한 `event_title`)
  5. `기한`: (기존 스크래핑한 `duration`)
  6. `내용`: (★중요★ 이벤트 참여 방법이나 조건, 대상 등 요약본)
  7. `카테고리`: (★중요★ 이미지 내 품목을 기반으로 `OTC`, `의약외품`, `비급여ETC`, `급여ETC`, `기타` 등 추출)
  8. `혜택`: (★중요★ 할인율, 추가증정품, 사은품 등 요약)
  9. `Thumbnail`: (로컬 저장된 썸네일 경로)
  10. `Detail`: (로컬 저장된 디테일 경로)

## 🙋 Open Questions (개발 진행 전 결정사항)
1. **Vision LLM API 모델 선택**: OpenAI, Google, Anthropic 중 어떤 회사의 API를 사용할 것인가?
2. **API Key 설정**: 선택한 서비스의 API 발급 및 `.env` 파일 등록 여부.
3. **처리 범위 조율**: 전체 40~50여 개의 이벤트 일괄 처리 시 소요되는 비용/시간을 고려해, 점진적/선별적으로 돌릴지에 대한 처리 방식 협의.
